## Problem Statement

Single AI agents building software hit two persistent walls: (1) context degradation and "context anxiety" on long tasks, and (2) self-evaluation failure where agents reliably praise their own mediocre output. Anthropic's research demonstrated a 3-agent architecture (Planner → Generator → Evaluator) produces categorically different output — not incrementally better, but in a different tier. No open-source framework packages this specific pattern with built-in telemetry to prove it works.

## What Makes This Cool

The adversarial tension between builder and critic — "getting to the truth agentically." An evaluator that resists the people-pleasing bias baked into every LLM, grades with calibrated skepticism, and produces telemetry that objectively proves whether the pattern works. Point it at a benchmark, walk away, come back to a side-by-side report showing solo-vs-trio with hard numbers.

## Constraints

- Linux-first, macOS supported (Docker optional with local-process fallback for non-Linux)
- Programming language agnostic — benchmarks span multiple languages
- 5 benchmark projects across a range of sizes: 3 small (15-30 min), 2 medium (60-90 min)
- Strict adherence to Anthropic article's methodology and claims
- Acceptance criteria cannot bend — every feature earned by proving correctness and quality
- Telemetry data (cost, score delta, iteration count) informs decisions about retry limits, evaluator strictness thresholds, and whether to skip the planner on small tasks
- Target audience: seasoned professionals wanting high velocity with no bugs and well-documented, well-architected PRs

## Premises

1. **The 3-agent pattern is tested across task sizes with strict adherence to the article's methodology** — the telemetry proves where it wins and where it doesn't. (Revised per Codex challenge: include small tasks to test where trio breaks even or loses.)
2. **Evaluator separation is the core lever**, not better self-prompting of a single agent.
3. **Telemetry can objectively prove quality** AND determine the right mix of orchestration.
4. **V1 = 5 distinct benchmark projects** (3 small at 15-30 min, 2 medium at 60-90 min), no human interaction, with telemetry showing taste/quality/correctness. Small benchmarks test where trio breaks even; medium benchmarks test where trio advantage emerges.
5. **Target: seasoned professionals** wanting high velocity with no bugs and well-documented PRs that don't generate tech debt.

## Approaches Considered

### Approach A: CLI Benchmark Harness (Chosen)
- `harnessa run <benchmark> --mode solo|trio`
- Hard-coded Planner→Generator→Evaluator→retry loop
- 5 canned benchmarks (3 small: bugfix/feature across Python/TS/Go, 2 medium: full-stack features)
- Solo-vs-trio comparison per benchmark
- JSON + markdown telemetry per run
- Effort: M (human: ~2 weeks / CC: ~2-3 hours)
- Risk: Low
- Pros: Ships fast, proves thesis with data, both cross-model reviewers recommended this
- Cons: Not pluggable yet, hard-coded prompts

### Approach B: Pluggable Framework with Benchmarks
- Full framework from PROJECT_SPEC.md — orchestrator, pluggable YAML criteria, file-based comms
- Effort: L (human: ~6 weeks / CC: ~6-8 hours)
- Risk: Medium — over-engineering before proof
- Cons: Longer to first result

### Approach C: Evaluator-First
- Build only the Evaluator + telemetry, run against existing solo output
- Effort: S (human: ~1 week / CC: ~1-2 hours)
- Risk: Low but incomplete — no end-to-end proof

## Recommended Approach

**Approach A: CLI Benchmark Harness** — with the critical constraint that acceptance criteria never bend. Each capability is earned by proving correctness and quality through the benchmark runs. The architecture can grow toward B over time, but every addition must be justified by telemetry showing it improves output.

**Stack (per Codex recommendation):** Python 3.12, Typer (CLI), Pydantic (schemas), LiteLLM (model-agnostic), pytest (testing), SQLite/JSONL (telemetry), Docker (benchmark isolation, optional — local-process fallback available).

## Benchmark Definitions (V1)

### Benchmark 1: Small Bugfix — Python CLI (15-30 min)
- **Repo:** A Python CLI tool (~500 LOC) with a known bug: argument parser silently drops flags containing `=` signs
- **Starting state:** Failing test case included, bug not yet fixed
- **Acceptance criteria:** Fix passes all existing tests + the failing test, no regressions, clean diff
- **Evaluator verdicts:** PASS if fix is minimal and correct; FAIL if fix introduces new edge cases or is overly broad

### Benchmark 2: Small Feature — TypeScript Library (15-30 min)
- **Repo:** A TypeScript utility library (~800 LOC) needing a new `retry()` function with exponential backoff
- **Starting state:** Interface defined in types, no implementation, test file with 12 test cases (all failing)
- **Acceptance criteria:** All 12 tests pass, handles edge cases (zero retries, negative delay, abort signal), documented
- **Evaluator verdicts:** PASS if all tests green + type-safe; FAIL if tests are modified instead of implementation

### Benchmark 3: Small Bugfix — Go HTTP Server (15-30 min)
- **Repo:** A Go HTTP API (~600 LOC) with a race condition in the connection pool under concurrent requests
- **Starting state:** Flaky test that reproduces ~40% of the time with `-race` flag
- **Acceptance criteria:** Race detector clean on 100 runs, no performance regression, minimal diff
- **Evaluator verdicts:** PASS if race-free + existing tests pass; FAIL if fix is a mutex around everything

### Benchmark 4: Medium Feature — Python Full-Stack (60-90 min)
- **Repo:** A FastAPI + SQLite TODO app (~2000 LOC) needing a complete "tags" feature: model, API endpoints, filtering, tests
- **Starting state:** Working app, no tags concept, 15 existing tests passing
- **Acceptance criteria:** 4 new endpoints, tag CRUD, filter-by-tag, 10+ new tests, migration, documentation updated
- **Evaluator verdicts:** PASS if feature works end-to-end and existing tests don't break; FAIL on stubs, missing migrations, or broken existing features

### Benchmark 5: Medium Feature — React + Node Full-Stack (60-90 min)
- **Repo:** A React + Express dashboard (~3000 LOC) needing a real-time notifications system: WebSocket server, notification bell UI, read/unread state, persistence
- **Starting state:** Working dashboard with auth, no notification infrastructure
- **Acceptance criteria:** WebSocket connection, notification creation API, bell component with unread count, mark-as-read, 8+ tests, no regressions
- **Evaluator verdicts:** PASS if notifications work in a real browser interaction test; FAIL on mock-only testing or broken existing functionality

## Solo Mode Specification

For fair A/B comparison, solo mode must be rigorously controlled:

- **Same model** as trio mode (same provider, same model ID, same temperature)
- **Same prompt content** — solo receives the original task description (identical to what the Planner receives in trio mode), but gets no spec expansion, no sprint contract, no evaluator feedback
- **Same token budget** — solo gets the same maximum token spend as the trio's total (planner + generator + evaluator combined)
- **Same wall-clock limit** — solo gets the same maximum duration
- **Same tools** — git, file I/O, shell, same Docker environment
- **No self-evaluation loop** — solo runs once, produces output, done (mirrors baseline from the article)
- **Same acceptance criteria** — graded by the same evaluator with the same criteria (evaluator runs post-hoc on solo output)

## Failure Modes & Recovery

| Failure | Detection | Recovery | Telemetry |
|---------|-----------|----------|-----------|
| Evaluator fails all criteria after max iterations (3) | Iteration counter hits limit | Mark sprint FAILED, log all scores and feedback, continue to next benchmark | `status: "max_iterations_exceeded"` |
| Model API error (rate limit, timeout, 500) | HTTP status / exception | Retry with exponential backoff: 3 retries, 10s/30s/90s delays. After 3 failures, mark run FAILED | `api_errors: [{type, retry_count, resolved}]` |
| Docker container hangs / OOM | 10-minute per-command timeout | Kill container, log last output, mark benchmark FAILED | `status: "timeout"`, `last_output: "..."` |
| Generator produces unparseable output | Schema validation on agent output | Re-prompt once with explicit format instructions. If still fails, mark FAILED | `parse_errors: [{agent, attempt, raw_output}]` |
| Evaluator rubber-stamps (all scores 9-10) | Score distribution check: if all criteria ≥ 9, flag as suspicious | Log warning, still accept result but tag telemetry with `suspicious_approval: true` | Human spot-check queue |
| Global run timeout | Wall-clock limit per benchmark (small: 45 min, medium: 120 min) | Graceful shutdown: save partial telemetry, mark INCOMPLETE | `status: "timeout"`, partial scores preserved |

## Goodhart Mitigation (V1)

The biggest risk is evaluator criteria being gameable — agents optimize the rubric instead of real quality. V1 mitigations:

1. **Test-suite validation (all benchmarks):** Evaluator runs the project's full test suite. Tests are pre-written in the benchmark repo and NOT visible to the generator during implementation — they are stored in a `_eval/` directory excluded from the generator's working tree and injected by the evaluator at grading time. Passing tests is necessary but not sufficient.
2. **Output fixture comparison (CLI/backend benchmarks):** For non-UI benchmarks, evaluator compares actual output against expected fixtures (deterministic input → expected output). This is the "Playwright equivalent" for backend work.
3. **Diff quality check:** Evaluator grades the git diff — minimal changes, no unnecessary file touches, no commented-out code, no debug artifacts left behind.
4. **Regression detection:** Evaluator runs the FULL existing test suite, not just new tests. Any regression = automatic FAIL regardless of other scores.
5. **Human calibration gates (per evaluator prompt version):** Success criteria #5-6 (false-positive rate < 20%, finds > 50% of real bugs) are validated by human spot-check once per evaluator prompt version, not per benchmark run. Per-run, we rely on the calibrated prompt.

**Hypothesis to validate:** For CLI/backend benchmarks, test-suite + fixture comparison is sufficient to prevent Goodhart collapse. If benchmarks show high scores but brittle code (detected in human spot-check), the evaluator criteria need Playwright-style interaction testing even for backend work.

## Telemetry Schema (per run)

Every run emits a JSON manifest capturing full provenance — including model identity:

```json
{
  "run_id": "uuid",
  "benchmark": "small-bugfix-python",
  "mode": "trio | solo",
  "model": {
    "provider": "anthropic",
    "model_id": "claude-opus-4.6",
    "temperature": 0.7,
    "max_tokens": 8192
  },
  "agents": {
    "planner":   { "model_id": "...", "tokens_in": 0, "tokens_out": 0, "duration_s": 0, "cost_usd": 0 },
    "generator": { "model_id": "...", "tokens_in": 0, "tokens_out": 0, "duration_s": 0, "cost_usd": 0 },
    "evaluator": { "model_id": "...", "tokens_in": 0, "tokens_out": 0, "duration_s": 0, "cost_usd": 0 }
  },
  "iterations": 2,
  "max_iterations": 3,
  "scores": { "product_depth": 7, "functionality": 8, "code_quality": 7 },
  "bugs_found": 3,
  "bugs_fixed": 2,
  "tests_passed": 27,
  "tests_failed": 0,
  "regressions": 0,
  "verdict": "PASS | FAIL",
  "suspicious_approval": false,
  "total_cost_usd": 1.24,
  "total_duration_s": 1200,

  "parse_errors": [],
```

Model identity is captured per-agent (agents may use different models) and at the run level. This enables cross-model comparison: "Does Opus outperform Sonnet enough to justify 5x cost?"

## Success Criteria

1. `harnessa run <benchmark> --mode trio` completes all 5 benchmarks without human intervention
2. `harnessa run <benchmark> --mode solo` completes same benchmarks as control
3. Telemetry shows where trio outperforms solo with hard numbers (scores, bugs caught, cost)
4. Telemetry shows where trio does NOT outperform solo (honest about limits)
5. Evaluator false-positive rate < 20% — **calibration gate**, validated by human spot-check once per evaluator prompt version
6. Evaluator finds > 50% of bugs a human reviewer would find — **calibration gate**, same validation cadence
7. Evaluator consistency: same artifact scored twice produces scores within ±1 per criterion
8. End-to-end variance: report score distributions across 3+ runs of same benchmark; don't gate on ±1 (generation is non-deterministic) but flag outliers
