Building a Critic That Actually Criticizes

  • 07
  • “Used an LLM to improve my argument over 4 hours… ask it to argue the opposite. LLM demolishes the entire argument. lol.” — Andrej Karpathy
  • { "functionality": 1, "verdict": "FAIL" } ↓ feedback loop { "functionality": 10, "verdict": "PASS" }
  • Anti-people-pleasing rules • Any test fails → functionality ≤ 4 • No new tests → test coverage ≤ 4 • Stub implementation → product depth ≤ 3 • All scores ≥ 7? Flag as suspicious
  • Calibration still imperfect: TS benchmark scored func=8 with 50% tests failing.

18

It’s Not Always Worth It

  • 18
  • Use Trio • Features touching 3+ files • Unfamiliar bugs with unclear root cause • Architecture refactors • Full-stack features • Anything where “does it actually work?” is uncertain
  • Skip It • Single-line fixes • Formatting or linting • Dependency bumps • Tasks where you already know the fix • Simple, well-understood changes
  • Rule of thumb: if a solo agent gets it right on the first try, trio is just overhead. The sweet spot is medium-complexity work at the edge of model capability.
19

What We Found That the Article Didn’t Cover

  • 19
  • 1 Planner value = structure, not scope The spec gave the generator a roadmap. On medium tasks that structure enabled correct first attempts; on small tasks it was overhead.
  • 2 Evaluator leniency persists Even with hard rules, we still saw func=8 with 50% tests failing. Guardrails help, but skepticism remains an active calibration problem.
  • 3 Telemetry keeps the claims honest Every run emits structured JSON. No “it feels better” hand-waving — if you can’t measure it, you can’t trust it.
